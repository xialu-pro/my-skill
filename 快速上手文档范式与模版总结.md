# 快速上手文档范式与模版总结

## 一、六种核心范式

| 范式 | 代表 | 适用场景 | 特点 |
|------|------|----------|------|
| **单页叙事型** | React | 概念密集型产品 | 一条故事线贯穿，Sandbox直接运行，无安装墙 |
| **向导步骤型** | Stripe、Docker | 集成型/工具型产品 | 严格编号步骤，"做完一件事"即完成 |
| **分流枢纽型** | Kubernetes | 受众多元、无单一路径 | 首屏自路由引导，按场景分流 |
| **极简安装型** | Rust | 工具链产品 | Install → Hello World → 离开，5分钟读完 |
| **索引目录型** | Fedora | 文档体系庞大、主题独立性强 | 纯目录索引，无引导分流，用户自行选择 |
| **分层索引-向导型** | NVIDIA CUDA | 路径极多（三层以上），无法用选择器简化 | 外层索引目录，内层编号步骤 |

## 二、范式对比矩阵

| 维度 | 单页叙事型 | 向导步骤型 | 分流枢纽型 | 极简安装型 | 索引目录型 | 分层索引-向导型 |
|------|------------|------------|------------|------------|------------|------------------|
| **代表** | React | Stripe/Docker | Kubernetes | Rust | Fedora | NVIDIA CUDA |
| **页面形态** | 单页长教程 | 分步向导 | 问题引导索引 | 极短单页 | 纯目录索引 | 外层目录+内层步骤 |
| **安装墙** | 无（Sandbox） | 有（第一步） | 无（跳过） | 有（单命令） | 各主题独立 | 各路径独立 |
| **个性化** | 无需 | 语言选择器 | 自路由问题 | OS自动检测 | 无引导 | 无引导（三层自选） |
| **代码示例** | Sandbox可运行 | 完整可运行 | 无 | 最小CLI | 各主题独立 | 各路径内编号步骤 |
| **离场设计** | 教程vs概念 | 上线清单 | 子章节链接 | 三种学习模态 | 自然浏览 | 无结构化离场 |
| **用户负担** | 低（一条路径） | 中（需选择） | 中（需回答问题） | 低（单命令） | 高（自行判断） | 极高（三层决策） |
| **首屏承诺** | 有（你将学到） | 有（完成后能） | 有（你是哪种用户） | 隐含（极短） | 无 | 无 |
| **预期输出** | Sandbox运行结果 | 每步骤后 | 无 | 有 | 各主题独立 | 无 |

## 三、分层索引-向导型 详细分析

### 结构特征

```
首屏（索引目录）
├─ 第一层选择：Windows / Linux
│   ├─ 第二层选择：Linux x86_64 / Linux ARM
│   │   ├─ 第三层选择：Redhat / Fedora / Ubuntu / Debian / ...
│   │   │   ├─ 第四层选择：RPM Installer / Runfile Installer
│   │   │   │   └─ 内层：编号步骤（1. 2. 3. ...）
```

### NVIDIA CUDA 结构示例

| 层级 | 决策内容 | 用户负担 |
|------|----------|----------|
| L1 | OS选择（Windows/Linux） | 低 |
| L2 | 架构选择（x86_64/ARM） | 中 |
| L3 | 发行版选择（Redhat/Fedora/Ubuntu/Debian/SLES/OpenSUSE/Amazon/WSL） | 高 |
| L4 | 安装方式选择（RPM/Runfile/Debian/Pip/Conda） | 高 |
| **总计** | **四层决策** | **极高** |

### 与索引目录型的关键差异

| 对比项 | 索引目录型（Fedora） | 分层索引-向导型（NVIDIA CUDA） |
|------|---------------------|-------------------------------|
| 外层结构 | 单层目录索引 | 多层目录索引（3-4层） |
| 内层结构 | 各主题独立内容 | 严格编号的向导步骤 |
| 内容形态 | 混合型（教程/FAQ/指南） | 统一的安装向导 |
| 一致性 | 低（各主题风格不一） | 高（所有路径结构相同） |
| 可复用性 | 低 | 高（步骤模板可复用） |

### 优缺点分析

**优点**：
- ✅ 覆盖全面：支持所有路径组合
- ✅ 结构一致：所有路径使用相同的编号步骤模板
- ✅ 可扩展：新增发行版只需复制步骤模板

**缺点**：
- ❌ 首屏无承诺：用户不知道"我能得到什么"
- ❌ 用户负担极高：需三层以上决策
- ❌ 无预期输出：步骤后缺少成功输出样本
- ❌ 无离场设计：结尾无结构化引导
- ❌ 无个性化引导：用户需自行判断选哪条路径

### 适用条件判断

```
Q: 是否需要采用分层索引-向导型？

判断条件（满足全部才适用）：
1. 路径维度 ≥ 3层（如：OS → 发行版 → 安装方式）
2. 路径总数 ≥ 10条（无法用选择器简化）
3. 所有路径的核心任务相同（都是"安装+验证"）
4. 各路径步骤结构高度相似（可复用模板）

如果路径维度 ≤ 2层 → 用向导步骤型 + 选择器
如果路径总数 ≤ 5条 → 用分流枢纽型或向导步骤型
```

## 六、五条核心规律（适用于所有范式）

| 规律 | 说明 | 模版示例 |
|------|------|----------|
| **规律1：首屏承诺结果** | 必须回答"我能得到什么" | `完成本指南后，你将能够……` |
| **规律2：个性化发生在第一个动作** | OS检测或语言选择器，后续内容个性化 | Rust OS自动检测 / Stripe语言选择器 |
| **规律3：代码示例四标准** | 可直接运行、有预期输出、最小可用、版本锁定 | 每个命令后附成功输出样本 |
| **规律4：安装墙是设计决策** | 能用Sandbox就推迟安装，否则"OS感知+单命令" | React无安装墙 / Rust单命令安装 |
| **规律5：结尾离场设计** | 按"然后呢"分流，不少于2条不超过4条路径 | 教程vs概念学习 / 读书/做题/看示例 |

## 八、openEuler范式选择建议

### 当前状态

| 维度 | openEuler现状 | 问题 |
|------|---------------|------|
| 当前范式 | 向导步骤型（有编号） | 基本正确 |
| 路径维度 | 2层（架构→安装方式） | 可简化 |
| 首屏承诺 | 无 | 需添加 |
| 预期输出 | 部分 | 需补充 |
| 离场设计 | 有但不够结构化 | 需优化 |

### 推荐范式

**向导步骤型**（类似Stripe/Docker），理由：
- 产品是操作系统，安装是核心体验
- 有明确目标：完成安装并登录
- 路径维度仅2层，可用选择器简化

### 改进建议

| 改进项 | 具体方案 |
